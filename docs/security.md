# Security

This project is a single-tenant local development build. The notes below
describe the existing controls and the controls that would be required
before exposing the API to real users.

## Implemented

### Authentication and authorization

- API runs behind a single hardcoded `local-user` identity used for
  every request. There is no authentication in the current build.
- All retrieval queries filter on `owner_id` so the data model is
  ready to enforce per-user isolation once auth is wired.
- Document delete and list endpoints both filter by `owner_id` before
  returning rows; another user's documents are never exposed even if
  the document UUID is guessed.

### Data at rest

- Object storage uses S3 server-side encryption when running against
  AWS S3 (default with `aws:kms` for the bucket policy).
- MinIO's default storage in local Docker is unencrypted on the host
  filesystem; this is acceptable for development data that is
  authored from scratch.
- Postgres data lives in a named Docker volume; encryption is left to
  the host OS in dev and to RDS storage encryption in production.

### Data in transit

- The local Docker network is private to the host; service-to-service
  traffic does not leave the machine.
- Production deployments terminate TLS at API Gateway / CloudFront.

### Logging and audit

- Every request gets a `request_id` (UUID v4) bound to the structlog
  context and echoed in the `X-Request-Id` response header.
- Logs include user id, query preview, and retrieved chunk ids — but
  not raw answer text by default — so PII exposure in logs is bounded.
- Logs are JSON to stdout; in production they ship to CloudWatch Logs
  and feed metric filters.

### Prompt-level safeguards

- The system prompt instructs the model to use only retrieved context,
  to cite every claim, and to abstain explicitly when context is
  insufficient.
- Citation enforcement is a separate post-processing step: short ids
  not present in retrieval are silently dropped from the response.

### Dependency hygiene

- All Python dependencies pinned by version range in `pyproject.toml`
  with a lockfile (`uv.lock`) generated by uv.
- Frontend pinned in `package.json`. Both layers are good candidates
  for `dependabot` or `renovate` in CI.

## Threat model

Using a STRIDE-lite breakdown.

### Spoofing

- *Risk:* Unauthenticated client poses as another user.
- *Status:* Open. v1 has no auth.
- *Production mitigation:* JWT-validated principal extracted from the
  `Authorization` header in `app.api.deps.current_user`. `owner_id`
  for all data writes and reads comes from this principal.

### Tampering

- *Risk:* Malicious user crafts a payload to inject SQL, path
  traversal in storage keys, or prompt injection into chat input.
- *Status:* SQLAlchemy parameterizes all queries; storage keys are
  generated server-side with `uuid4()`; user input is wrapped inside a
  `<context>` block in the prompt that the model is instructed to
  treat as data, not instructions.
- *Production mitigation:* Add an explicit prompt-injection
  classifier on user-supplied document text before it reaches the
  retrieval index; consider Constitutional AI guards on the chat
  endpoint.

### Repudiation

- *Risk:* User denies asking a question or uploading a document.
- *Status:* Every API call lands in the structured log with
  `request_id` and `user_id`. Documents have a `created_at` stamp and
  a stable storage key. The combination is sufficient for forensics.

### Information disclosure

- *Risk 1:* Cross-user data leak via retrieval.
- *Mitigation:* `owner_id` filter at the query layer; integration
  test `test_filters_by_owner` asserts this.
- *Risk 2:* The model regurgitates context that the requesting user
  isn't authorized to see.
- *Mitigation:* Retrieval is owner-scoped, so the model only ever
  receives chunks belonging to the requester.
- *Risk 3:* PII or secrets land inside an indexed document.
- *Production mitigation:* Run AWS Comprehend's PII detection on
  document text during ingestion; flag or redact before indexing.

### Denial of service

- *Risk:* Malicious user uploads enormous files or floods chat.
- *Status:* No rate limiting in v1.
- *Production mitigation:* WAF rate limits at CloudFront / API Gateway;
  per-user quota in the application; max upload size in the multipart
  parser.

### Elevation of privilege

- *Risk:* User accesses admin endpoints or other tenants' data.
- *Status:* No role-based access in v1.
- *Production mitigation:* Cognito groups + a route-level RBAC
  decorator; admin endpoints behind a separate role check.

## Secrets handling

- AWS credentials live only in `.env` (gitignored).
- No secrets are committed to the repo. `.env.example` contains only
  placeholder values.
- In production, AWS Secrets Manager or SSM Parameter Store provides
  database credentials and any third-party API keys. The application
  reads them via the standard AWS SDK chain — no code changes needed.

## Known limitations

- No CSRF protection on document upload; relies on same-origin policy
  in the browser. A hostile site embedding the upload endpoint cannot
  read responses, but could trigger a multipart upload if a session
  exists. CSRF tokens are required before public deployment.
- No content-type sniffing on uploads; we trust the client-supplied
  content type. Production should `python-magic`-check the bytes.
- Document delete only removes from S3 best-effort. A failure here
  leaves the file orphaned in storage; the DB row is removed
  regardless. A production version would use S3 lifecycle rules or a
  tombstone table to reconcile.
